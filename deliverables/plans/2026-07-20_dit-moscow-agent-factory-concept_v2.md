## 1. Назначение документа

Документ описывает техническую концепцию городской платформы создания и управления
ИИ-агентами и обосновывает ключевые архитектурные решения в сравнении с альтернативами.
Речь идёт о новой системе, проектируемой и создаваемой совместно с ДИТ под задачи и
требования города: технологический стек, состав компонентов и порядок развёртывания —
предмет совместного проектирования. ООО «Окстех» вносит в проект архитектурный подход и
опыт промышленной эксплуатации собственной мультиагентной платформы (раздел 11); сама
платформа не переносится — город получает систему, спроектированную для него и
принадлежащую ему.

## 2. Целевая архитектура

### 2.1. Состав

Платформа состоит из пяти функциональных компонентов с разделёнными зонами ответственности:

| Компонент | Ответственность |
|---|---|
| **Реестр и конструктор** | Каталог агентов и их спецификаций, комнаты, права, конструктор специальностей, интерфейсы администрирования. |
| **Фабрика (конвейер)** | Сборка агентов из спецификаций, песочница, аттестация, управление жизненным циклом и версиями. |
| **Агенты** | Исполнители: универсальное ядро + специализация конфигурацией. |
| **Слой данных (MCP)** | Единый контролируемый доступ агентов к городским ИС, документам, справочникам. |
| **Шлюз моделей** | Единая точка доступа к языковым моделям с политиками выбора, лимитами и учётом. |

```
Каналы (веб, мессенджеры, почта, внешние ассистенты)
   │ A2A
   ▼
Оркестратор ──A2A──► агенты-специалисты (в комнатах)
   │                        │
   │ MCP                    │ MCP
   ▼                        ▼
Слой данных: инструменты ──► городские ИС, документы, календари…
                            │
                            ▼ единый API
                     Шлюз моделей ──► модели города / российские / open-source
```

### 2.2. Почему именно такое членение

Границы компонентов совпадают с границами ответственности эксплуатирующих команд. Это даёт
три практических следствия. Во-первых, независимое развитие: слой данных расширяется новыми
инструментами без пересборки агентов; шлюз моделей обновляется без остановки процессов;
реестр меняет правила без вмешательства в код исполнителей. Во-вторых, заменяемость: каждый
компонент имеет описанный интерфейс, поэтому любой из них может быть доработан или заменён
другой командой — в том числе без участия ООО «Окстех». В-третьих, прозрачную модель
безопасности: весь доступ агентов к данным проходит через один слой, который можно проверять
и ограничивать централизованно, вместо десятков прямых интеграций «агент → система», каждая
из которых была бы отдельной поверхностью атаки.

Ядро платформы публикуется в открытом коде на Mos.Hub. Для города это не жест, а механизм:
код доступен для аудита службам информационной безопасности, городские команды и подрядчики
дорабатывают платформу без лицензионных барьеров, а накопленная функциональность не исчезает
при смене исполнителя.

## 3. Модель агента: универсальное ядро + декларативная спецификация

### 3.1. Устройство

Каждый агент состоит из двух частей.

**Ядро** — одинаково у всех агентов: сервер протокола A2A с карточкой возможностей, клиент
слоя данных (MCP), аутентификация входящих вызовов, ограничение частоты запросов, эндпоинты
здоровья и метрик, сквозная трассировка действий.

**Спецификация** — то, что делает агента «специалистом»: роль (системный промпт — см.
раздел 6 о языке шаблонов), навыки, перечень доступных инструментов, список разрешённых
агентов-контрагентов, комнаты, назначенная модель и лимиты.

### 3.2. Возможности, которые это открывает

- **Специальность за дни, а не месяцы** — новый агент — например, проверяющий комплектность
  заявок по свежему регламенту — описывается в конструкторе и собирается из готового ядра.
  Ведомство получает работающего специалиста без проектной разработки, тендера на подрядчика
  и интеграционного проекта.
- **Массовые обновления без риска рассинхронизации** — исправление в ядре (безопасность,
  протоколы, логирование) распространяется на весь парк агентов одной пересборкой. При
  «зоопарке» отдельных разработок то же исправление потребовало бы множества независимых
  проектов с разными сроками и качеством.
- **Регламенты меняются без релизов** — роли и правила хранятся в реестре, версионируются и
  применяются без переразвёртывания кода. Правка регламента ведомства — операция методолога,
  а не разработчика.

### 3.3. Технические гарантии

Конфигурация централизована: агент получает её из реестра при старте и не стартует без него
(принцип fail-fast). Это исключает два классических отказа распределённых систем — дрейф
конфигураций (у всех агентов гарантированно актуальные права и правила) и «осиротевшие»
экземпляры со старыми полномочиями. Изменение спецификации автоматически инициирует
пересборку агента: работающий код и его описание в реестре не могут разойтись — свойство,
критичное для аудита.

## 4. Протокольный слой: A2A + MCP

### 4.1. Разделение взаимодействий

Платформа разделяет два принципиально разных типа взаимодействий и использует для каждого
профильный открытый стандарт:

- **MCP (Model Context Protocol)** — «вертикаль»: доступ агента к инструментам и данным.
  Вызовы по описанным схемам, единый слой над городскими системами, белый список
  инструментов на каждого агента.
- **A2A (Agent-to-Agent)** — «горизонталь»: взаимодействие агентов между собой.

### 4.2. Что даёт A2A (спецификация v1.0.1, Linux Foundation, Apache 2.0)

- **Задача с жизненным циклом** — вместо одиночного вызова — задача с состояниями (принята /
  в работе / требуется ввод / завершена). Процессы, идущие часами и днями — проверка пакета
  документов с уточнениями, согласование в несколько шагов — выполняются штатно, а не через
  обходные решения с таймаутами. Исполнитель может задать встречный вопрос и дождаться
  ответа; заказчик видит промежуточный прогресс.
- **Карточка возможностей (Agent Card)** — стандартизованное самоописание агента для
  автоматического поиска исполнителей. Внутренняя логика, данные и промпты не раскрываются
  (принцип Preserve Opacity). Для города это основа межведомственного взаимодействия без
  слияния контуров: агент одного ведомства пользуется услугами агента другого, не получая
  доступа к его внутренностям.
- **Три режима взаимодействия** — синхронный, потоковый и асинхронный режимы покрывают весь
  спектр задач — от быстрой справки до многодневного процесса.
- **Нейтральное управление и 6 SDK** — протокол передан в Linux Foundation — ставка на него
  не привязывает город ни к одному вендору. Шесть официальных SDK (Python, Go, JavaScript,
  Java, .NET, Rust) означают, что агентов смогут писать городские команды, подрядчики и
  университеты на привычном стеке.

### 4.3. Почему не «всё через MCP»

Распространённая альтернатива — оборачивать агентов в MCP-инструменты и вызывать как
функции. Авторы спецификации A2A называют этот путь принципиально ограничивающим;
конкретные потери:

| Свойство | Агент как MCP-инструмент | Агент по A2A |
|---|---|---|
| Жизненный цикл задачи | Нет: вызов завершился или упал по таймауту. | Состояния, прогресс, длительные задачи. |
| Встречный вопрос исполнителя | Не предусмотрен. | Состояние «требуется ввод». |
| Обнаружение исполнителей | Список функций сервера. | Карточки + реестр по навыкам. |
| Состояние процесса | На вызывающей стороне. | У задачи. |
| Семантика для аудита | Инструмент и автономный агент неразличимы. | Разделены по протоколам. |

Последняя строка важнее, чем кажется: когда в журнале аудита «вызов инструмента» и
«поручение автономному агенту» выглядят одинаково, невозможно построить осмысленную политику
контроля. Разделение протоколов делает журнал читаемым: записи MCP — работа с данными,
записи A2A — передача ответственности между агентами.

Для коротких синхронных сценариев MCP-обёрток достаточно; проигрыш проявляется на
многошаговых процессах с согласованиями — основном классе задач ведомств.

## 5. Комнаты: организация пространства агентов

### 5.1. Устройство

Комната — это курируемый реестр карточек агентов, совмещённый с областью действия прав.
Вход агента в комнату означает: запись в реестре комнаты, выдачу доступов к её агентам и
допусков к её инструментам. Видимость данных и исполнителей ограничена границей комнаты.
Универсальные агенты (чтение документов, подготовка текстов, календарь) состоят в нескольких
комнатах одновременно.

Соответствие стандарту: спецификация A2A описывает три стратегии обнаружения агентов и прямо
рекомендует курируемые реестры для корпоративных сред с требованиями к управляемости. API
реестра стандартом не зафиксирован — реестр «комнат» является легитимным расширением
стандарта, а не отклонением от него.

### 5.2. Возможности и выгоды

- **Подключение без интеграционного проекта** — новый агент, добавленный в комнату,
  автоматически становится доступен остальным: они находят его по карточке. Не требуется ни
  правка конфигураций существующих агентов, ни новые сетевые доступы точка-точка.
- **Границы данных = границы комнат** — модель прав повторяет организационную структуру:
  финансовая комната не видит кадровую. Для аудита ИБ ответ на вопрос «кто имеет доступ к
  данным X» — это список участников одной комнаты, а не разбор матрицы интеграций.
- **Межведомственные процессы** — ведомства делегируют отдельных агентов в общие процессные
  комнаты, не раскрывая свои внутренние контуры.
- **Защита межагентных вызовов** — обязательные токены, опциональная криптографическая
  подпись запросов с защитой от повторного воспроизведения, ограничение частоты вызовов.

## 6. Язык шаблонов промптов

### 6.1. Проблема

Роль агента задаётся системным промптом. В наивной реализации это монолитный текст: общие
правила города, требования ведомства и особенности конкретной задачи перемешаны в одной
«простыне». Последствия: одно и то же правило копируется в десятки промптов и неизбежно
расходится по версиям; правка общего требования означает ручную правку всех агентов;
сравнить две версии роли и понять, что изменилось по существу, — трудно; ответственность
методолога и инженера не разделена.

### 6.2. Предлагаемое решение: специализированный язык шаблонов на базе JSON

Промпт описывается структурой с переменными, которые можно переиспользовать:

```json
{
  "rules":  "Всегда выбирай красное",
  "prompt": "Система должна следовать следующим правилам: {rules}"
}
```

Переменной может быть константа, отдельное правило или целый повторяющийся набор инструкций.
Спецификация агента ссылается на общие блоки, а не копирует их:

```json
{
  "city_tone": "@city/style/official-tone",
  "doc_rules": "@city/rules/document-handling",
  "vedomstvo": "Проверка отчётности по формам 0503117, 0503737",
  "prompt":    "Ты — агент-контролёр. {city_tone} {doc_rules}
                Твоя специализация: {vedomstvo}"
}
```

Уровни каталога переменных: платформа (общегородские правила) → комната (правила области) →
агент (локальные параметры). Нижний уровень может переопределять верхний по явно описанным
правилам.

### 6.3. Возможности и выгоды

- **Единая правка** — общегородское правило (тон обращений, запреты, формат протокола
  проверки) меняется в одном месте каталога — и применяется ко всем агентам, использующим
  переменную. Сегодня это десятки ручных правок с риском пропустить агента.
- **Композиция ролей** — специальность = базовый шаблон + отраслевые наборы правил +
  параметры ведомства. Новые роли собираются из проверенных блоков, а не пишутся с нуля.
- **Правки как управляемые изменения** — структура сравнивается и версионируется: правка
  регламента становится просматриваемым изменением конкретной переменной с историей, автором
  и согласованием — а не переписыванием текста, в котором изменения не видны.
- **Валидация до выпуска** — схема позволяет автоматически проверять, что все переменные
  заполнены и ссылки разрешаются; конвейер аттестации (раздел 8) прогоняет изменённый шаблон
  по эталонным заданиям — правка правила не попадёт в работу, если ломает поведение агента.
- **Разделение труда** — методологи ведомств управляют значениями правил на понятном им
  языке; инженеры платформы — структурой шаблонов и механикой подстановки. Это снимает
  главный конфликт наивного промпт-инжиниринга, где любая правка требует и предметных, и
  инженерных компетенций одновременно.

Проработка синтаксиса (формат ссылок, области видимости, наследование, каталог переменных в
реестре) — задача этапа совместного проектирования; в отрасли есть прецеденты шаблонизации
промптов, которые будут проанализированы при выборе решения.

## 7. Оркестрация процессов

Оркестратор — отдельный агент, ведущий составные задачи:

- **Планирование** — по описанию задачи и реестру доступных исполнителей строится план шагов
  с зависимостями.
- **Параллельное волновое исполнение** — шаги с выполненными зависимостями запускаются
  одновременно. Практический смысл: проверка пакета из независимых документов идёт
  параллельно, а не последовательно — время процесса определяется самым долгим шагом, а не
  суммой всех.
- **Устойчивость к частичному отказу** — сбой одного шага не валит процесс: зависимые шаги
  корректно помечаются пропущенными, независимые продолжаются, результат содержит явную
  картину «что выполнено, что нет и почему».
- **Верификация результата** — после выполнения — проверка качества с оценкой уверенности и
  ограниченным репланированием: сомнительный результат перепроверяется, а не молча
  передаётся дальше.
- **Событийная модель** — события оркестрации публикуются в реестр — на них строится живой
  граф «кто кого привлёк» и полная трассировка процесса. Для разбора инцидента доступна вся
  цепочка: постановка, план, исполнители, промежуточные результаты.

Выгода для ведомства формулируется просто: процесс, а не переписка. Сотрудник ставит
задачу — платформа сама подбирает исполнителей, координирует их, контролирует качество и
возвращает результат с полной историей выполнения.

## 8. Конвейер фабрики и модель качества

### 8.1. Путь агента

```
Заявка ведомства → Сборка (конструктор) → Песочница (эталонные задания)
   → Аттестация (агент-аттестатор + человек) → Выпуск в комнату
```

Ни один агент не получает доступ к данным города, минуя этот путь. Песочница —
изолированная среда на тестовых данных (production, тестовые и отраслевые контуры
разделены). Аттестация — прогон эталонных заданий специальным агентом-аттестатором,
работающим на самой сильной доступной модели, с протоколом проверки; финальный допуск
подтверждает человек.

### 8.2. Показатель зрелости

Каждый агент и навык несёт показатель зрелости: прототип → песочница → аттестован →
production → ветеран. Показатель хранится в реестре и виден при выборе исполнителя.

Зачем это городу. Во-первых, управление риском: ведомство видит, что можно доверить
агенту-«ветерану», а что — только аттестованному под присмотром. Во-вторых, масштабирование
без потери качества: тот же конвейер проходят навыки внешних команд — университетских,
подрядных. Качество расширений гарантируется процессом (эталонные задания, аттестация), а не
доверием к конкретному исполнителю — это единственная модель, при которой открытая
экосистема не превращается в свалку непроверенных решений.

## 9. Модельный слой, контроль и наблюдаемость

- **Шлюз моделей** — агенты не содержат кода конкретных провайдеров; смена модели —
  изменение конфигурации, а не переписывание агентов. Политика выбора фиксируется на
  сценарий: модели города — основной контур; российские коммерческие и локальные open-source
  — по задаче; передовые зарубежные — точечно, в разрешённых сценариях без персональных
  данных (например, аттестация агентов). Город использует лучшую доступную модель для каждой
  задачи и меняет решения по мере развития рынка — без заложника в виде переписывания парка
  агентов.
- **Учёт и лимиты** — расход токенов считается по каждому вызову и агрегируется по контурам;
  действуют бюджеты на задачу, лимиты на пользователя, бюджеты контуров. Затраты на ИИ
  становятся управляемой статьёй с прозрачной себестоимостью процесса, а не непредсказуемым
  счётом в конце месяца.
- **Наблюдаемость** — по каждому запросу фиксируется траектория: шаги, вызовы инструментов
  и агентов, длительность, статус, расход. Любой результат можно разобрать до отдельного
  действия — основа и для аудита ИБ, и для методической работы над качеством агентов.
- **Human-in-the-loop** — значимые действия — отправка документов, изменение данных —
  требуют подтверждения человеком на уровне конкретного действия, а не «одобрения сессии»
  целиком. Автоматизация наращивается постепенно: сначала агент готовит, человек
  подтверждает; по мере накопления доверия (и зрелости агента) доля подтверждений снижается
  осознанным решением ведомства.
- **Развёртывание** — полностью в контуре города, контейнерное; автоматизированная сборка и
  безопасный редеплой с верификацией запуска.

## 10. Экосистема: университеты и живая среда

### 10.1. Задача шире технической

Платформа со временем потребует всё больше навыков, инструментов и агентов — больше, чем
может произвести любой отдельный подрядчик. Поэтому целью проекта является не только
техническая среда, но и экспертное сообщество вокруг неё — живая среда, в которой навыки
создаются, проверяются и совершенствуются многими независимыми участниками по общим
правилам.

### 10.2. Роль университетов

- **Написание навыков и ролей по оплачиваемым заказам** — ведомства публикуют потребности на
  витрине; университетские команды берут заказы и сдают результат через общий конвейер
  аттестации. Это устойчивая экономика участия, а не разовые конкурсы: поток навыков не
  иссякает после хакатона.
- **Порог входа снижен языком шаблонов** — значительная часть работы над навыком — правила,
  инструкции, эталонные задания — доступна студентам-предметникам (юристам, экономистам,
  документоведам), а не только программистам (раздел 6). Университеты подключают к платформе
  не одну кафедру, а несколько факультетов.
- **Учебный процесс на реальных задачах** — курсовые и дипломные работы — это навыки для
  конкретных ведомств с измеримым результатом аттестации; лучшие работы попадают в
  production и остаются в портфолио автора с показателем зрелости.
- **Кадровый резерв города** — студенты, прошедшие путь «навык → песочница → аттестация →
  production», к выпуску знают городские процессы и платформу — это готовый канал найма для
  ДИТ и ведомств.
- **Научная составляющая** — методики оценки качества агентов, эталонные наборы заданий,
  исследования на обезличенных траекториях — предмет совместных работ и публикаций.

### 10.3. Механика живой среды

Сообщество держится на инфраструктуре, а не на энтузиазме: открытое ядро и документация на
Mos.Hub; SDK протокола на шести языках — команды работают на привычном стеке; общедоступная
песочница с эталонными заданиями — порог входа измеряется днями; витрина с публичным
показателем зрелости навыков и репутацией авторов — качество видно, хорошая работа
накапливает вес; интеграция с действующими городскими программами (хакатон «Лидеры Цифровой
Трансформации», Академия инноваторов, программы Московского инновационного кластера) —
приток новых участников встроен в существующие процессы.

### 10.4. Что это даёт городу

Мощность фабрики перестаёт быть ограничена ресурсом одного поставщика — она растёт с
сообществом. Экспертиза по агентным системам накапливается в городе: в ведомствах
(методологи), в университетах (авторы навыков), в командах ДИТ (владельцы платформы). И это
же — долгосрочная защита проекта: система с живым сообществом развивается и переживает смену
подрядчиков; система одного вендора живёт ровно столько, сколько интерес вендора.

## 12. Сравнение с альтернативными подходами

### 12.1. Аренда закрытой платформы

Наиболее заметное предложение на рынке — GigaCowork («Салют для бизнеса»); сравнение по
открытым данным (тестовый доступ с 19.05.2026):

| Критерий | GigaCowork | Городская фабрика |
|---|---|---|
| Код | Закрытый, поставщика. | Открытое ядро на Mos.Hub, принадлежит городу. |
| Контур | Облако поставщика; on-premise не заявлен. | Полностью в периметре города. |
| Модели | Экосистема одного вендора. | Политика выбора на сценарий, включая модели города. |
| Межагентный протокол | Не заявлен (публично известны MCP-интеграции). | A2A v1.0.1 (Linux Foundation). |
| Создание агентов | Self-service в чате. | Конвейер: песочница, аттестация, зрелость. |
| Аудит | Не раскрыт. | Трассировка каждого вызова, учёт расхода. |
| Развитие | По плану вендора. | По правилам города: витрина и внешние команды (6 SDK). |

Подход самообслуживания оптимизирует скорость создания агента отдельным сотрудником.
Предлагаемая архитектура оптимизирует другое: управляемость парка агентов организации —
качество, права, аудит, стоимость. Для города, отвечающего за процессы ведомств и данные
жителей, второй набор свойств является определяющим.

### 12.2. No-code конструкторы

Автоматизируют личную рутину сотрудника; в них нет модели прав между подразделениями,
аттестации и обособленных контуров данных — задачи организации они не решают.

### 12.3. Заказная разработка «агент под задачу»

Каждый агент — отдельный проект со своим стеком, циклом сопровождения и интеграциями
точка-точка; стоимость каждого следующего агента не снижается. В фабрике предельная
стоимость новой специальности — конфигурация плюс аттестация; интеграции переиспользуются
через общий слой данных, а правила — через язык шаблонов.

### Итог

Предлагаемая архитектура одновременно: (1) оставляет код, данные и модели в собственности и
периметре города; (2) опирается на открытые стандарты под нейтральным управлением — без
привязки к вендору, включая ООО «Окстех»; (3) даёт процессный контроль качества и
пооперационный аудит; (4) масштабируется рынком исполнителей и переиспользованием (ядро,
инструменты, шаблоны правил), а не мощностями одного поставщика.
