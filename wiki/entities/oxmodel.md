# Oxmodel

LLM-шлюз [[oxtech-platform]]: единая точка обращения агентов к моделям. Агенты не содержат
провайдеро-специфичного кода — весь трафик идёт через OpenAI-совместимый эндпоинт шлюза.

## Что это на самом деле

⚠️ **Oxmodel — не собственная разработка Oxtech.** Это внешний open-source AI-шлюз **GOModel**
(github.com/ENTERPILOT/GOModel), развёрнутый Docker-образом `enterpilot/gomodel:v0.1.35`.
Исходников в workspace нет; в репозитории лежит только systemd-юнит запуска контейнера.
«Oxmodel» — внутреннее брендирование; в коде такое название не встречается вообще, там
`gomodel` / `OXTECH_AI_*`.

Это не порок — использовать зрелый open-source шлюз разумно. Но говорить о нём как о
собственном компоненте платформы перед заказчиком рискованно: при технической проверке
расхождение вскроется.

| Параметр | Значение |
|---|---|
| Хост | `agents.oxteam.me:8094`, base path `/gomodel` |
| Наружу | `https://agents.oxteam.me/gomodel/*` (nginx, rate-limit 20 r/s) |
| Хранилище | PostgreSQL `gomodel_prod4` + Redis (кэш ответов) |
| API | OpenAI-совместимый: `/v1/chat/completions`, `/v1/models`, embeddings |
| Аутентификация | master key + managed keys `sk_gom_*`, привязанные к `user_path`, SHA256 в БД шлюза |

## Провайдеры: заявлено и реально

Презентация (слайд 7) заявляет: «GigaChat, YandexGPT — для контуров с локализацией; OpenAI,
Anthropic, Gemini — где допустимо» и «поддержка локальных open-source моделей».

| Провайдер | В презентации | В коде Oxtech | В конфиге шлюза |
|---|---|---|---|
| OpenAI | да | через шлюз | ключ настроен |
| Anthropic | да | обёртка удалена | ключ настроен |
| DeepSeek | — | обёртка удалена | ключ настроен |
| Gemini | да | обёртка удалена | **ключ не настроен** |
| **GigaChat** | да, как ключевой аргумент | **кода нет** | **не настроен** |
| **YandexGPT** | да, как ключевой аргумент | **кода нет** | **не настроен** |
| Локальные модели | да | нет | не настроено |

⚠️ **GigaChat и YandexGPT в production не работают ни в каком виде.** Провайдеро-специфичные
обёртки (`gigachat_model.go`, `yandex_model.go`, `anthropic_model.go`, `deepseek_model.go`,
`google_gemini_model.go`) удалены из `oxtech_agent` в задаче 207. Остались только декоративные
следы: строки в валидаторе конфига, проброс `GIGACHAT_API_KEY` в env, который никто не читает,
и GigaChat в устаревшей прайс-таблице. Прямых эндпоинтов Сбера и Яндекса в коде нет.

Существенная оговорка: GOModel — внешний продукт, и поддерживает ли он GigaChat/YandexGPT
апстримом, из этого workspace проверить нельзя. Установленный факт — в **развёрнутой
конфигурации Oxtech они не подключены**, и переключение на них не проверялось.

**Фактически все агенты сейчас ходят в OpenAI.** Алиасы моделей заданы в админке шлюза (не в
git): `oxtech_lite → gpt-5.4-mini`, `oxtech_middle → gpt-5.2`, `oxtech_pro → gpt-5.4`,
`oxtech_embeddings → text-embedding-3-small`. Админу Portal в UI видны только три
обезличенных алиаса — реальные имена моделей скрыты.

Риск для встречи с ДИТ Москвы: локализация моделей и импортозамещение — вероятный вопрос.

### Статус российских моделей

Шлюз архитектурно провайдеро-независим и OpenAI-совместим, поэтому подключение российских
моделей — работа по конфигурации, а не переписывание агентов. Формулировка «архитектурно
поддерживаем любого провайдера, сейчас работаем на OpenAI» точна; «поддерживаем GigaChat» — нет.

Ландшафт российских LLM и роли провайдеров (GigaChat/YandexGPT — кандидаты, MWS GPT — аналог) —
[[russian-llm-landscape]]; та страница приведена в соответствие с кодом (провайдеры помечены
как «не подключён»), расхождения между страницами нет.

## Выбор модели

Трёхуровневая схема: Portal (`AgentConfig#model_config`, jsonb) → env-переменные
`OXTECH_AI_BASE_URL` / `OXTECH_AI_KEY` → фабрика модели в агенте. Без обеих переменных агент
не поднимает модель вообще (`ErrGatewayNotConfigured`) — fail-fast.

Ключ выбирается по типу агента: `procurement` получает ключ с `user_path=/oxtech/keylink`,
остальные — `/oxtech/agents`. Другие пути: `/oxtech/portal`, `/oxtech/documents`,
`/oxtech/embeddings` — это даёт раздельный учёт расхода по контурам.

## Учёт токенов — реализован полноценно

Самый достоверный блок из заявленных. `usage` парсится из ответа шлюза (в стриминге —
принудительно через `StreamOptions{IncludeUsage: true}`) и расходится по четырём приёмникам:

1. Prometheus — `oxtech_llm_tokens_by_type_total` с лейблами agent_id/agent_type/model/token_type.
2. Траектория запроса — `tokens_input` / `tokens_output` (см. [[observability]]).
3. Ответ ADK — `UsageMetadata`.
4. Оценка стоимости — `oxtech_llm_cost_estimate_cents`.

Хранение в Portal: `agent_trajectories` (агент + пользователь + сессия — самая гранулярная
привязка), `agent_metrics_stats` (агрегаты по агенту и периоду), `system_metrics_stats`
(общесистемные, включая cache_read/cache_write).

**Лимиты реализованы, три независимых механизма:**

| Механизм | Где | Параметры |
|---|---|---|
| Пер-задачный бюджет | `internal/tokenbudget` | atomic-счётчик, при превышении — событие `budget_exceeded` в стрим |
| Пер-пользовательский часовой лимит | `internal/ratelimit`, Redis sorted-set | скользящее окно 1 ч, дефолт **200 000 токенов/час**, предупреждение на 80% |
| Бюджеты по `user_path` | админка GOModel | вне репозитория |

Индивидуальный лимит пользователя приходит из Portal через метаданные A2A; перехват — на уровне
ADK-колбэка `RateLimitGuard` перед вызовом LLM.

⚠️ Прайс-таблица оценки стоимости устарела: содержит `gpt-4o`, `gemini-2.0-flash`, `GigaChat-2`,
но не содержит ни алиасов `oxtech_*`, ни `gpt-5.*`. Для текущих production-моделей метрика
стоимости молча не наполняется — то есть «учёт стоимости» из слайда о наблюдаемости сейчас
не работает.

## Отказоустойчивость

- **Fallback между моделями в коде Oxtech отсутствует.** Заявлен как функция самого GOModel
  (задача 142), проверить нельзя — исходников шлюза нет. При ошибке LLM вызов в агенте просто
  падает наверх.
- Ретраи — только для пустого ответа (`len(choices) == 0`), до 3 попыток. Сетевые ошибки, 5xx
  и 429 **не ретраятся**.
- ⚠️ **Таймаута на вызов агент → шлюз нет**: HTTP-клиент создаётся без поля `Timeout`. Зависший
  ответ шлюза удерживает горутину агента до отмены контекста извне.
- Portal → шлюз: таймаут 25 с (bulkhead после инцидента с исчерпанием тред-пула Puma).
  Стриминговые пути админ-дашборда отбрасываются с 501 по той же причине.

## On-premise

Архитектурно достижимо: шлюз — Docker-образ, ставится в контур заказчика; агенты обращаются к
нему по env-переменной. Это подтверждает тезис презентации. Но конкретно развёртывание в чужом
контуре с локальными моделями не выполнялось и в коде не зафиксировано.

Ландшафт доступных в РФ моделей, кандидаты на подключение (MWS GPT, Cotype, T-Pro) и
ближайший функциональный аналог (MWS GPT) — [[russian-llm-landscape]].

Предшественник идеи — провайдеро-независимость в [[ox-ai-workers]] (смена LLM через `uri_base`).

См. также [[observability]].
