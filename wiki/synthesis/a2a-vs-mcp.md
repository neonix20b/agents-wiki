# A2A vs MCP: разница подходов и ограничения

Анализ для позиционирования [[oxtech-platform]] (A2A+MCP) против архитектур, где всё взаимодействие построено на [[mcp]], включая межагентное. Источники — [[a2a-vs-mcp-web-research]].

## Суть протоколов

| | [[mcp]] (Anthropic, 11.2024) | [[a2a]] (Google, 04.2025) |
|---|---|---|
| Направление | «Вертикальное»: агент → инструменты и данные | «Горизонтальное»: агент ↔ агент |
| Модель | Клиент-сервер, вызов функций по схеме | Пиринговая: client agent ↔ remote agent |
| Единица работы | Tool call: запрос → ответ | Task с жизненным циклом (в работе / нужен ввод / завершена) |
| Discovery | Список инструментов сервера | Agent Card — самоописание возможностей агента |
| Прозрачность внутренностей | Сервер раскрывает схему инструментов | Агент — «чёрный ящик», раскрывает только возможности |

Консенсус индустрии: протоколы **комплементарны** — MCP расширяет «что может один агент», A2A — «как агенты сотрудничают».

## Ограничения подхода «всё через MCP» (агент как инструмент)

Когда агента оборачивают в MCP-инструмент, межагентное взаимодействие сводится к синхронному вызову функции. Следствия:

1. **Нет жизненного цикла задачи** — вызов либо завершился, либо нет; долгие задачи упираются в таймауты, стандартного механизма «задача принята, работаю, вот прогресс» нет.
2. **Нет встречных вопросов** — вызванный агент не может штатно запросить уточнение у вызвавшего (в A2A это состояние input-required); диалог надо эмулировать поверх протокола.
3. **Нет discovery возможностей** — MCP описывает функции, а не компетенции агента; Agent Card в A2A позволяет оркестратору выбирать исполнителя по способностям.
4. **Состояние на вызывающем** — контекст многошаговой совместной работы приходится таскать через параметры вызовов.
5. **Иерархия вместо кооперации** — только «вызвал → получил результат»; параллельная работа нескольких агентов с обменом промежуточными результатами не выражается естественно.
6. **Семантика**: инструмент детерминирован по контракту, агент — нет; смешение уровней усложняет мониторинг, права и аудит (кто действует: инструмент или автономный агент?).

## Что честно признать
- Подход «всё через MCP» проще: один протокол, меньше движущихся частей; для коротких синхронных сценариев («подготовь документ по шаблону») его достаточно.
- Преимущество A2A проявляется на **сложных, длительных, многоагентных процессах** — что совпадает с целевым сегментом Oxtech ([[gigacowork-vs-oxtech]]).

## Позиционирование Oxtech
Oxtech поддерживает **оба протокола по назначению**: MCP — доступ к данным и инструментам (60+), A2A — координация агентов ([[orchestration]]); каждый A2A-вызов виден в трассировке (`a2a_call: financer, writer, mailer`). Это архитектурное решение, а не просто «+1 протокол»: слой координации отделён от слоя доступа к данным, что даёт масштабируемость команды агентов и корректный аудит.

⚠️ Про [[gigacowork]]: утверждение «у них всё взаимодействие, включая межагентное, построено на MCP» — рабочая гипотеза (публично заявлены только MCP-коннекторы к системам; межагентный протокол не раскрыт). Перед использованием в публичных материалах — проверить по документации тестового доступа.
