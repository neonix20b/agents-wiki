# Концепция для ДИТ Москвы: городская фабрика ИИ-агентов

Рабочая концепция предложения. Контекст: встреча с ДИТ назначена, технология заинтересовала;
мотивация ДИТ — избежать зависимости от Сбера (внедрение [[gigacowork]] означает вендор-лок) и
получить собственную платформу, превосходящую GigaCowork. Канал подачи: прямая встреча +
оформление через [[moscow-innovation-cluster]] (Техлаб). Формат deliverable — презентация.

## Идея в одном абзаце

Город получает не продукт вендора, а **собственную фабрику ИИ-агентов**: небольшое
технологическое ядро (конструктор агентов) + понятные правила его расширения. Код агента
универсален ([[oxtech-platform]]: один бинарник, разные build tags/конфигурации) — агенты
различаются набором инструментов, навыков и ролей. Взаимодействие — открытые протоколы:
[[a2a]] между агентами + discovery, [[mcp]] к данным. Ядро публикуется как open source на
Mos.Hub — город владеет технологией, Oxtech — технологический партнёр ядра.

## Ключевые элементы

### 1. Конструктор (ядро)
Универсальный код агента + декларативная спецификация: роль (промпт — см.
[[prompt-template-language]]), навыки, инструменты
(MCP), права (allowlist), комнаты. Новая «специальность» = конфигурация, не разработка.
Основа уже есть: единая кодовая база агентов Oxtech, конфигурация из Portal, сборка с build
tags ([[oxtech-manager]]).

### 2. Комнаты = рынки навыков (метафора городских площадей)
Комната — пространство, где агенты публикуют услуги (Agent Card) и находят друг друга
(discovery). Площадей несколько: по ведомствам, по процессам. Универсальные агенты
(Reader, Writer, Reviewer) состоят сразу в нескольких комнатах. Вход в комнату = права
доступа к её данным и агентам. Техбаза: A2A Agent Cards + реестр агентов; discovery по
комнатам — доработка (сейчас allowed_agents статичен, см. [[orchestration]]).

### 3. Фабрика / «роддом» агентов
Конвейер порождения: заявка ведомства → сборка из конструктора → песочница → аттестация →
выпуск в комнату. **Показатель зрелости агента** (уровни: прототип → песочница → аттестован →
production → ветеран) с автоматической оценкой специальным агентом-аттестатором на самой
сильной доступной модели + контрольные прогоны на эталонных заданиях. Развитие идеи
Reviewer из [[agents-roster]].

### 4. LLM-стратегия: осознанный выбор модели на сценарий
Через OpenAI-совместимый шлюз ([[oxmodel]] / GOModel) подключается микс: дообученные модели
самого ДИТ (LoRA, их платформа «ИИ» — см. [[russian-llm-landscape]]), российские коммерческие,
локальные open-source, и **точечно Anthropic/OpenAI в разрешённых сценариях** (например,
аттестация зрелости агентов — без ПДн и служебных данных). Правило: выбор модели — явное
решение per-сценарий, зафиксированное в конфигурации агента; полный учёт токенов по контурам
уже реализован.

### 5. Витрина инструментов и экономика университетов
Витрина: каталог MCP-инструментов и навыков с версионированием и статусом зрелости.
Ведомства публикуют потребности; университетские команды берут **оплачиваемые заказы** на
разработку навыков/ролей (модель рынка, не хакатон-разовость). Аттестация чужих навыков —
через ту же фабрику. Синергия: ЛЦТ, Академия инноваторов, КИТ.

### 6. Целевая боль: формализация неформализованного
Большая часть рутины ведомств — перенос неформализованных данных в формы: бухгалтерия,
проверка контрольных соотношений между документами. **Пилот: контрольные соотношения** —
агент читает пакет документов (Reader), сверяет соотношения (Reviewer/Financer), формирует
протокол расхождений (Writer). Измеримый эффект: время проверки, % найденных расхождений.

## Роли сторон
- **ДИТ** — владелец инсталляции и данных, оператор платформы, владелец процесса аттестации.
- **Oxtech** — архитектор и maintainer ядра (open core на Mos.Hub), обучение команд, ревью
  расширений, SLA-поддержка, enterprise-компоненты.
- **Университеты** — поставщики навыков через витрину (оплачиваемые заказы).

## Позиционирование против GigaCowork — прямое сравнение
Слайд-таблица «собственная платформа vs аренда у вендора»: владение кодом (open source vs
закрытая экосистема), данные и модели в контуре города vs облако Сбера, любые LLM осознанно
vs GigaChat, развитие своими силами + рынок университетов vs roadmap вендора, A2A+MCP vs
MCP-центричная архитектура ([[a2a-vs-mcp]]), аттестация и зрелость агентов vs «навыки за
минуты». Тезис: «GigaCowork автоматизирует сотрудника; городская фабрика создаёт
управляемую популяцию агентов, которой город владеет».

## Принцип подачи: опыт ≠ предложение (Д. Смолев, 2026-07-20)

**Для ДИТ создаётся новая система совместно с городом, а не копия платформы Oxtech.**
В материалы для ДИТ нельзя переносить архитектуру и цифры действующей системы как
«предлагаемое решение» — из нашего продукта берётся только опыт. Правильная структура
любого документа/презентации для ДИТ:

1. **Предлагаемое решение** — самостоятельная целевая архитектура, спроектированная под
   задачи города (без стека, внутренних цифр и деталей нашей инсталляции; стек и детали —
   предмет совместного проектирования с ДИТ).
2. **Опыт ООО «Окстех»** — отдельный раздел: действующая production-платформа, её цифры
   (147 MCP-инструментов, единая кодовая база агентов, учёт токенов, трассировка),
   продукты OxDocs/Keylink, свидетельство RU 2026613582 — как доказательство
   реализуемости подхода и компетенции команды, не как продаваемый артефакт.

Причина: заказчик хочет владеть новой системой, построенной под него; подача чужой
системы как готового решения противоречит самой идее фабрики и open core.

## Честные оговорки (внутренние, не для слайдов)
1. Российские LLM в нашем production сейчас не подключены — для пилота с моделями ДИТ
   потребуется работа по шлюзу ([[oxmodel]], ⚠️ известное расхождение).
2. Discovery по комнатам, витрина, аттестатор зрелости — новые компоненты; в ядре есть
   базис (Agent Cards, реестр, tiers), но это roadmap, а не текущая функциональность.
   В презентации отделять «работает сегодня» от «создаём вместе с городом».
3. Для госзакупки вероятен вопрос реестра российского ПО Минцифры — статус неизвестен
   ([[intellectual-property]]); свидетельство RU 2026613582 есть, это плюс.
4. Заявленные цифры презентации («деплой 15–25 с», blue-green) кодом не подтверждены —
   в материалах для ДИТ использовать только защитимые формулировки (см. oxtech-platform).

## Следующие шаги
1. Презентация 10–15 слайдов (deliverable) на базе этой концепции.
2. Проверить статус в реестре российского ПО; при отсутствии — запустить через МИК.
3. Прототип discovery по комнатам и метрики зрелости — к демо для ДИТ.
