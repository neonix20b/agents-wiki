# A2A (Agent-to-Agent)

Открытый протокол взаимодействия между агентами (Google, апрель 2025). «Горизонтальный»:
агент ↔ агент, роли client/remote agent, Agent Cards для discovery возможностей, задачи с
жизненным циклом, длительные задачи и промежуточный прогресс.

## Реализация в Oxtech

| Параметр | Значение |
|---|---|
| SDK | `a2aproject/a2a-go/v2` **v2.3.1** |
| Фреймворк агентов | ADK Go **v1.4.0** |
| Версия протокола | **1.0** |
| Транспорт | JSON-RPC 2.0 over HTTP + SSE |
| Agent Card | `/.well-known/agent-card.json`, публично, без авторизации |

Миграция с v0.3.13 на v2.3.1 выполнена задачей 186, все 11 production-агентов проверены.

⚠️ Документация отстала: `docs/3_protocols/a2a.md` заявляет a2a-go v0.3.10 и ADK v1.2.0,
`oxtech_agent/CLAUDE.md` — v0.3.6 и v0.6.0.

## Имена методов: ловушка, стоившая production-бага

В a2a-go v2 методы называются **CamelCase без слеша**, а не как в классическом описании
спецификации:

| Wire-метод (реальность) | Классическое имя | Статус |
|---|---|---|
| `SendMessage` | `message/send` | основной путь |
| `SendStreamingMessage` | `message/stream` | основной путь, SSE |
| `GetTask` | `tasks/get` | используется для polling |
| `CancelTask` | `tasks/cancel` | реализован, не вызывается |
| `ListTasks`, `SubscribeToTask` | `tasks/list`, `tasks/resubscribe` | не вызываются |
| push-уведомления | `tasks/pushNotificationConfig/*` | **не работают** |

Задача 204: Ruby-клиент Portal слал `message/send`, агенты отвечали `-32601 method not found`,
веб-чат показывал заглушку «Ответ получен». Исправлено, но документация до сих пор описывает
старые имена — следующий разработчик наступит туда же.

Push-уведомления не реализованы: обработчик создаётся без опций, поэтому все push-методы
возвращают `ErrPushNotificationNotSupported`. Карточка это честно отражает.

## Что публикует агент

Имя, описание, версия, интерфейс (URL + JSON-RPC + версия протокола), режимы ввода-вывода
(`text/plain`, `application/json`), `Capabilities: {Streaming: true}`, схемы безопасности
(apiKey в заголовке `Authorization`, опционально HMAC), провайдер `Oxtech / oxteam.me`.

⚠️ Карточка **статична** — строится один раз при старте. Изменение конфигурации в Portal не
отразится в карточке без перезапуска агента. Skills декоративны — см. [[agents-roster]].

## Аутентификация между агентами

Три слоя на входящем HTTP:

1. **Bearer-токен** (обязательный), сравнение constant-time. Публичны только `/health`,
   `/metrics` и Agent Card (последняя — явно по спецификации A2A). Токены выдаёт Portal:
   в конфигурацию каждого агента подмешиваются `base_url` и `auth_token` разрешённых агентов.
   ⚠️ Токен симметричный: один и тот же `AUTH_TOKEN` используется и для входящих A2A-вызовов,
   и для исходящих обращений агента в Portal MCP.
2. **HMAC-SHA256** (опциональный, включается наличием `A2A_HMAC_SECRET`): подпись
   `HMAC(secret, "<ts>.<body>")`, anti-replay по дрейфу часов ≤ 5 минут.
3. **Rate limiting**: 120 запросов/мин на ключ авторизации + посуточный токен-лимит на
   пользователя.

mTLS, OAuth и JWT между агентами не используются.

## Отказоустойчивость

Философия — **fail-soft на discovery, fail-fast на вызове**.

| Сценарий | Поведение |
|---|---|
| Агент не подключён | немедленная ошибка без сетевого вызова |
| Агент остановлен между вызовами | refresh раз в 30 с обнаруживает пропажу и удаляет из реестра |
| Agent Card недоступна | предупреждение + синтетическая карточка, подключение продолжается |
| Карточка отдаёт `localhost` | URL принудительно перезаписывается внешним из конфига |
| Streaming упал | fallback на `SendMessage`; если и он упал — ошибка наверх |
| Агент пересоздан, токен протух | при смене URL перечитывается конфиг из Portal со свежими токенами |

⚠️ Ретраев, backoff и circuit breaker на уровне A2A-транспорта нет (circuit breaker подключён
только к MCP-клиенту). Таймаута у HTTP-клиента намеренно нет — дедлайн задаёт контекст, чтобы
не рвать SSE. Серверный лимит на задачу — 3 минуты, бюджет 50 000 токенов.

⚠️ Задачи хранятся в памяти и не переживают перезапуск агента: после рестарта `GetTask` по
ранее выданному `task_id` задачу не найдёт.

В метриках и траекториях фактические A2A-вызовы видны как `a2a_call` — см. [[observability]].
