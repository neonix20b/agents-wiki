# A2A (Agent-to-Agent)

Открытый протокол взаимодействия между агентами. «Горизонтальный»: агент ↔ агент, роли
client/remote agent, Agent Cards для discovery возможностей, задачи с жизненным циклом,
длительные задачи и промежуточный прогресс.

## Статус протокола

Протокол малоизвестен, поэтому при разговоре с заказчиком его приходится представлять.
Факты, которые стоит называть (сверено с официальными материалами 2026-07-20 —
[[a2a-protocol-official-docs]]):

| Параметр | Значение |
|---|---|
| Управление | **Linux Foundation**, вклад Google, лицензия Apache 2.0 |
| Версия спецификации | **v1.0.1** от 28.05.2026, 11 релизов |
| Транспорт | JSON-RPC 2.0 поверх HTTP(S) + SSE |
| SDK | 6 языков: Python, Go, JavaScript, Java, .NET, Rust |

Ключевое для госзаказчика: протокол **не принадлежит вендору**. Он передан в Linux Foundation
и развивается сообществом под Apache 2.0. Ставка на A2A — не привязка к Google, в отличие от
ставки на закрытую экосистему одного поставщика.

## Преимущества протокола

### 1. Непрозрачность как модель безопасности

Заявленная цель — «Preserve Opacity»: агенты сотрудничают, **не раскрывая** внутреннюю логику,
память и проприетарные инструменты. Наружу агент публикует только описание того, что умеет,
а не то, как он это делает.

Для организации со многими подразделениями это принципиально: агент ведомства может работать
с агентом другого ведомства, не открывая ему ни свои данные, ни свои правила. Ровно это
позволяет строить «комнаты» из [[dit-agent-factory-concept]], не сводя всё в один общий контур.

### 2. Задача с жизненным циклом, а не вызов функции

Единица работы — **Task** с состояниями: принята, в работе, требуется ввод, завершена.
Отсюда следует то, чего нельзя получить от обычного вызова инструмента:

- длительные процессы не упираются в таймаут запроса;
- агент может штатно **запросить уточнение** и дождаться ответа (состояние input-required);
- виден промежуточный прогресс, а не только финальный результат.

Для государственных процессов, которые идут днями и требуют согласований, это не деталь,
а необходимое условие.

### 3. Три режима взаимодействия из коробки

| Режим | Когда нужен |
|---|---|
| Синхронный (`sendMessage`) | короткий запрос-ответ |
| Потоковый (SSE, `sendMessageStream`) | видеть ход выполнения по мере работы |
| Асинхронный (push-уведомления) | долгие задачи, клиент отключается и возвращается позже |

Принцип проектирования сформулирован авторами как **asynchronous-first**: длительные задачи и
разрывы соединения поддержаны нативно, а не эмулируются поверх.

### 4. Совместимость между фреймворками

Агенты, написанные на разных технологиях, работают вместе без индивидуальных интеграций —
шесть официальных SDK. Практический смысл для модели «фабрики»: команды-подрядчики
(например, университетские) пишут агентов на своём стеке, а не на навязанном, и агент всё
равно встраивается в общую среду.

### 5. Обнаружение агентов — три штатные стратегии

- **Well-known URI** — карточка по `/.well-known/agent-card.json` (RFC 8615).
- **Курируемые реестры** — каталог карточек с запросом по навыкам и провайдеру; спецификация
  прямо называет это подходящим для корпоративных сред с требованиями к управляемости.
- **Прямая конфигурация** — заданные заранее адреса.

⚠️ Существенно и полезно: **API реестра спецификацией пока не стандартизован.** Значит,
собственный реестр «комнат» — легитимное расширение поверх стандарта, а не отклонение от него.
Это снимает возражение «вы придумываете своё вместо стандарта».

### 6. Корпоративная безопасность заложена в спецификацию

Аутентификация и авторизация, расширенные карточки только для авторизованных клиентов
(Authenticated Extended Cards), mTLS и OAuth 2.0 на эндпоинтах, выборочная выдача карточек
реестром по правам клиента, рекомендация против статических секретов. Плюс мультимодальность:
текст, файлы, структурированный JSON.

### Как это звучит одной фразой

Обёртывание агента в обычный инструмент авторы называют «fundamentally limiting»: теряется
автономия и способность рассуждать в многошаговом взаимодействии. A2A сохраняет и то и другое.
Развёрнутый разбор ограничений архитектур «всё через MCP» — [[a2a-vs-mcp]].

## Имена методов: ловушка, стоившая production-бага

В a2a-go v2 методы называются **CamelCase без слеша**, а не как в классическом описании
спецификации:

| Wire-метод (реальность) | Классическое имя | Статус |
|---|---|---|
| `SendMessage` | `message/send` | основной путь |
| `SendStreamingMessage` | `message/stream` | основной путь, SSE |
| `GetTask` | `tasks/get` | используется для polling |
| `CancelTask` | `tasks/cancel` | реализован, не вызывается |
| `ListTasks`, `SubscribeToTask` | `tasks/list`, `tasks/resubscribe` | не вызываются |
| push-уведомления | `tasks/pushNotificationConfig/*` | **не работают** |

Задача 204: Ruby-клиент Portal слал `message/send`, агенты отвечали `-32601 method not found`,
веб-чат показывал заглушку «Ответ получен». Исправлено, но документация до сих пор описывает
старые имена — следующий разработчик наступит туда же.

Push-уведомления не реализованы: обработчик создаётся без опций, поэтому все push-методы
возвращают `ErrPushNotificationNotSupported`. Карточка это честно отражает.

## Что публикует агент

Имя, описание, версия, интерфейс (URL + JSON-RPC + версия протокола), режимы ввода-вывода
(`text/plain`, `application/json`), `Capabilities: {Streaming: true}`, схемы безопасности
(apiKey в заголовке `Authorization`, опционально HMAC), провайдер `Oxtech / oxteam.me`.

⚠️ Карточка **статична** — строится один раз при старте. Изменение конфигурации в Portal не
отразится в карточке без перезапуска агента. Skills декоративны — см. [[agents-roster]].

## Аутентификация между агентами

Три слоя на входящем HTTP:

1. **Bearer-токен** (обязательный), сравнение constant-time. Публичны только `/health`,
   `/metrics` и Agent Card (последняя — явно по спецификации A2A). Токены выдаёт Portal:
   в конфигурацию каждого агента подмешиваются `base_url` и `auth_token` разрешённых агентов.
   ⚠️ Токен симметричный: один и тот же `AUTH_TOKEN` используется и для входящих A2A-вызовов,
   и для исходящих обращений агента в Portal MCP.
2. **HMAC-SHA256** (опциональный, включается наличием `A2A_HMAC_SECRET`): подпись
   `HMAC(secret, "<ts>.<body>")`, anti-replay по дрейфу часов ≤ 5 минут.
3. **Rate limiting**: 120 запросов/мин на ключ авторизации + посуточный токен-лимит на
   пользователя.

mTLS, OAuth и JWT между агентами не используются.

## Отказоустойчивость

Философия — **fail-soft на discovery, fail-fast на вызове**.

| Сценарий | Поведение |
|---|---|
| Агент не подключён | немедленная ошибка без сетевого вызова |
| Агент остановлен между вызовами | refresh раз в 30 с обнаруживает пропажу и удаляет из реестра |
| Agent Card недоступна | предупреждение + синтетическая карточка, подключение продолжается |
| Карточка отдаёт `localhost` | URL принудительно перезаписывается внешним из конфига |
| Streaming упал | fallback на `SendMessage`; если и он упал — ошибка наверх |
| Агент пересоздан, токен протух | при смене URL перечитывается конфиг из Portal со свежими токенами |

⚠️ Ретраев, backoff и circuit breaker на уровне A2A-транспорта нет (circuit breaker подключён
только к MCP-клиенту). Таймаута у HTTP-клиента намеренно нет — дедлайн задаёт контекст, чтобы
не рвать SSE. Серверный лимит на задачу — 3 минуты, бюджет 50 000 токенов.

⚠️ Задачи хранятся в памяти и не переживают перезапуск агента: после рестарта `GetTask` по
ранее выданному `task_id` задачу не найдёт.

В метриках и траекториях фактические A2A-вызовы видны как `a2a_call` — см. [[observability]].
