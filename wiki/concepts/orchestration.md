# Оркестрация агентов

Паттерн: центральный оркестратор распределяет работу между специализированными агентами и
контролирует результат. Ключевой тезис Oxtech: модель отвечает → агент выполняет задачу →
**система агентов ведёт процесс**.

Эволюция в продуктах Oxtech:
- [[ox-ai-workers]]: `Assistant::Orchestrator` с заданным workflow + инструмент Pipeline для
  связи ассистентов и общего контекста.
- [[oxtech-platform]]: Orchestrator как отдельный агент; связь по открытому протоколу [[a2a]],
  инструменты через [[mcp]]; живой граф вызовов, трассировка каждого шага.

## Как это работает в коде

**Планирование.** LLM-планировщик получает текстовый список доступных агентов вида
`- Name (agent_type): Description` и возвращает JSON-план из шагов с зависимостями. Отдельного
маршрутизатора по skill'ам из Agent Card нет — выбор исполнителя целиком на LLM по текстовому
описанию агента.

**Параллельное выполнение — есть.** Волновая модель: планировщик отдаёт все шаги, чьи
зависимости выполнены, они запускаются одновременно, на границе волны — ожидание. Поддержан
частичный отказ: волна валит план, только если провалились **все** её шаги; шаги, зависящие
от упавших, каскадно помечаются `skipped`.

**Сборка результатов.** Один успешный шаг — результат как есть; несколько — конкатенация.
Результаты зависимостей подмешиваются в промпт следующего шага текстом.

**Верификация и репланирование.** После выполнения — LLM-верификация результата с
confidence-скором, порог 0.7, до одного репланирования и до двух ретраев на шаг.
⚠️ Верификация fail-open: при ошибке или непарсящемся ответе верификатора подставляется
`{OK: true, Confidence: 0.5}`, то есть сомнительный результат проходит.

**Синхронность.** Внутри оркестратора всё синхронно и стримингом — шаг блокируется до конца
SSE-потока. Асинхронный путь с `task_id` существует только на границе MCP-конвертера: вызов
orchestrator извне возвращает `task_id`, клиент опрашивает `agent_task_status`. Встроенного
цикла ожидания нет — polling на стороне вызывающего.

**События.** 10 типов событий оркестрации уходят в Portal через MCP-инструмент
`deliver_orchestration_event` — это то, что подсвечивает граф вызовов в админке.

## Известные дефекты

Существенны для демонстрации: если во время показа один из агентов остановлен, оркестрация
может повести себя некрасиво.

1. **Планировщик не видит реальный статус агентов.** Список доступных агентов у оркестратора —
   статический снимок конфига, снятый при старте, тогда как транспортный реестр обновляется
   каждые 30 с. Промпт при этом прямо утверждает LLM: «перечисленные агенты — это ВСЕ агенты
   системы, их статус: running». Последствие: остановленный агент остаётся в списке → шаг
   падает на `remote agent not connected` → три ретрая по мёртвому адресу → реплан по тому же
   списку. Обратное тоже верно: агент, поднявшийся динамически, но отсутствующий в
   `allowed_agents`, оркестратору невидим.
2. **`currentUserID` — общее мутируемое состояние синглтона.** При двух параллельных
   оркестрациях идентификатор пользователя в событиях может указать не на того адресата, а
   ранний `defer` обнулит ID у ещё работающей задачи.
3. **Гонки данных в выполнении плана**: статус шага и счётчик завершённых обновляются из
   горутин без блокировки. Требуется прогон `go test -race`.
4. **Streaming предпочитает partial поверх final**: при непустом частичном тексте финальный
   ответ молча отбрасывается — если поток чанков не покрыл весь ответ, результат окажется
   усечённым.

Пункты 1 и 2 зафиксированы в памяти рабочего пространства как известные, но не исправленные.
