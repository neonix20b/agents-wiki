# Наблюдаемость

Каждый агент [[oxtech-platform]] под наблюдением: видно, что он делал, кого вызывал, сколько
это стоило по токенам. Для организации это ключевое требование к автономным агентам —
проверяемость каждого действия.

## Траектории запросов

По каждому запросу записывается полная траектория. Хранение 14 дней, просмотр в админке с
фильтрами по статусу, транспорту, пользователю.

| Поле | Что содержит |
|---|---|
| transport | канал: telegram / web / mattermost / a2a |
| query | исходный запрос пользователя |
| steps (jsonb) | все шаги: тип (`tool_call` / `llm_call` / `context_load` / `thinking`), вход, выход, длительность, ошибка |
| tokens_input / tokens_output | расход токенов |
| total_duration_ms | длительность |
| status | success / error / timeout |

## Иерархическая трассировка

Помимо траекторий — дерево span'ов вызовов, 8 типов: `request`, `a2a_call`, `mcp_call`,
`llm_call`, `context_load`, `response`, `auth`, `tool_call`. Видно, кто кого вызвал:
`a2a_call: financer, writer, mailer`. Хранение 2 недели.

## Метрики агента

31 метрика Prometheus, онлайн-передача в Portal по WebSocket (ActionCable), агрегация по
часам/дням/неделям. Ключевые:

| Группа | Метрики |
|---|---|
| Межагентные | A2A-вызовы всего и по методам |
| Инструменты | MCP-вызовы всего и по инструменту, время выполнения |
| Модель | запросы, токены по типам и моделям (input/output/cache), оценка стоимости |
| Ресурсы | память, goroutines, размер бинарника |
| Надёжность | ошибки инструментов и модели, retry, состояние circuit breaker |
| Каналы | сообщения Telegram и веб-чата |
| Приватность | счётчик замаскированных PII |

## PII-маскирование

6 типов персональных данных: телефон, email, ИНН, СНИЛС, паспорт, номер карты. Маскируются в
ответах модели, логах и телеметрии. Включается на агента, набор типов настраивается.

## Аудит действий

Действия агентов логируются (тип действия, цель, метод, результат, длительность, пользователь,
сессия). Изменения ключевых сущностей (пользователи, письма, факты, транзакции, счета, события)
версионируются с атрибуцией через отдельный механизм аудита.

## Что это даёт заказчику

- **Полная трактовка запроса**: контекст, каждый вызов, ответ, токены — по любому обращению.
- **Прозрачность стоимости**: расход токенов виден по агенту, пользователю, модели.
- **Разбор инцидентов**: по траектории и дереву span'ов восстанавливается, что именно
  происходило и где сбой.

Учёт токенов и лимиты — см. [[oxmodel]]. Подтверждение значимых действий — [[hitl]].
Ограничение инструментов на агента — [[mcp]].
