# Конвейер подготовки документов (агент Writer)

Агент [[agents-roster|Writer]] не пишет документ одним запросом к модели. Он **декомпозирует
задачу на суб-агентов**, каждый из которых отвечает за свой узкий этап. Это оркестрация внутри
одного агента — уровнем ниже межагентной [[orchestration]].

Значение для позиционирования: декомпозиция делает задачу каждого шага узкой и проверяемой,
что повышает эффективность на **слабых моделях** (см. ниже). Это переносимое инженерное
know-how, а не только свойство конкретного продукта.

## Четыре суб-агента

Реализованы как ADK-суб-агенты внутри Writer-бинарника (build tag `document`).

| Суб-агент | Этап | Что делает |
|---|---|---|
| DocumentPlanner | plan | собирает контекст (поиск по данным, почта), выбирает шаблон, при необходимости задаёт уточняющие вопросы, формирует план и файлы задач по разделам, отдаёт план **человеку на одобрение** |
| DraftCoordinator (WriterPool) | write | читает план, режет на задачи-разделы, запускает параллельных писателей глав |
| ReviewCoordinator | review | по каждому черновику обращается к внешнему агенту [[agents-roster|Reviewer]], крутит цикл доработки |
| AssemblyCoordinator | assemble | детерминированно склеивает одобренные черновики, собирает LaTeX, компилирует в PDF |

Состояния конвейера: `planning → plan_review → writing → reviewing → assembling → compiling →
completed`. Состояние и артефакты (план, задачи, черновики, ревью, собранный документ) хранятся
в S3; [[oxdocs]] выступает REST-бэкендом этих артефактов.

## Параллельное написание глав

Внутри WriterPool каждая глава пишется **отдельным воркером своим отдельным вызовом модели**,
воркеры одной пачки идут параллельно. По умолчанию 5 воркеров (настраивается `DOCUMENT_WORKERS`),
план ограничен 8 задачами. Это ускоряет подготовку длинного документа и, что важнее, изолирует
каждую главу в собственный узкий запрос.

## Почему это помогает слабым моделям

Ключевой тезис. Писателю главы ставится максимально узкая, «заземлённая» задача: писать
**только** свою секцию, опираясь **исключительно** на переданный план и описание задачи, без
внешних знаний, без преамбулы и выводов, без метаданных.

Механизм эффекта:

- **Узкое пространство вывода.** Одна большая задача даёт слабой модели много способов ошибиться
  — потерять структуру, пропустить раздел, придумать факт. Узкая задача сужает пространство и
  делает результат предсказуемым.
- **Пошаговая проверяемость.** Каждый этап верифицируется отдельно: план одобряет человек,
  черновики проверяет Reviewer, компиляцию — гейт до 3 попыток. Ошибка ловится на своём шаге,
  а не всплывает в финальном документе.
- **Заземление против галлюцинаций.** Запрет опираться на внешние знания снимает главный риск
  слабой модели в фактологическом документе.

Чем слабее модель, тем больше выигрыш от декомпозиции: сильная модель вытянет монолитную задачу,
слабой — декомпозиция даёт то структурирование, которого ей не хватает.

**Замер.** Эффект подтверждён замером команды на lite-моделях (2026-07-20). GigaChat и DeepSeek
относятся к тому же классу слабых моделей — эффект переносится на них по классу.

⚠️ Две честные границы для технического разговора:
- Замер выполнен на lite-моделях; на GigaChat и DeepSeek он **переносится по принадлежности к
  классу**, прямого замера именно на них пока нет. Численные результаты в репозитории не
  зафиксированы — стоит оформить отдельно.
- Внутри одного Writer-агента **модель одна на все этапы** — поэтапного выбора («сильная на
  план, слабая на главы») сейчас нет. Разные модели по ролям достижимы только разными деплоями
  агентов (Reviewer — уже отдельный агент со своей моделью). Это возможное развитие, не текущая
  функция.

GigaChat/DeepSeek как провайдеры — статус подключения см. [[oxmodel]].

## Контроль качества встроен в конвейер

- **План — на одобрение человека.** Генерация глав не стартует, пока план не одобрен; гейт
  защищён на уровне кода, даже если модель попытается делегировать раньше времени.
- **Цикл доработки ревью.** Черновик с оценкой ниже порога переписывается и отправляется на
  повторное ревью, до 3 попыток; непрошедшее эскалируется человеку, но не блокирует сборку.
- **Компиляционный гейт.** Сборка LaTeX детерминирована (без модели), модель привлекается только
  для точечной починки ошибок компиляции, до 3 попыток.

## Два режима

- **Quick Create** — мгновенная заготовка документа по шаблону, без конвейера и без написания
  моделью. Для простых случаев.
- **Full Pipeline** — описанная выше схема из четырёх суб-агентов. Для содержательных документов.

Маршрутизацию между ними по интенту делает Communicator.
