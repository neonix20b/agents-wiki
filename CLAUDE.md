# agents-wiki — схема базы знаний

База знаний и рабочих документов по мультиагентной платформе Oxtech (oxteam.me).
Построена по паттерну «LLM Wiki» (Karpathy): LLM инкрементально ведёт и поддерживает
структурированную wiki поверх неизменяемых исходников. Человек курирует источники и
задаёт вопросы; LLM делает всю рутину — конспекты, кросс-ссылки, индексацию, поддержку
консистентности.

Язык содержимого — русский. Имена файлов и папок — английский, kebab-case.

## Структура

```
agents-wiki/
├── CLAUDE.md        # этот файл — схема и правила (слой «схема»)
├── index.md         # каталог всех страниц wiki — обновляется при каждом инжесте
├── log.md           # хронологический журнал операций (append-only)
├── inbox/           # входящие файлы, ожидающие обработки (drop zone)
├── raw/             # исходники — НЕИЗМЕНЯЕМЫ, только чтение
│   └── assets/      # изображения и вложения к исходникам
├── wiki/            # база знаний — ведёт LLM, человек читает
│   ├── sources/     # конспект каждого исходника (1 файл = 1 источник)
│   ├── entities/    # сущности: продукты, агенты, компании, люди
│   ├── concepts/    # концепции и технологии: A2A, MCP, оркестрация…
│   └── synthesis/   # обзоры, сравнения, выводы, ответы на вопросы
├── deliverables/    # рабочие документы (исходящие артефакты)
│   ├── contracts/   # договоры
│   ├── plans/       # планы
│   ├── reports/     # отчёты
│   ├── presentations/ # презентации
│   └── letters/     # письма и КП
└── templates/       # шаблоны deliverables
```

## Три слоя

1. **raw/** — источник истины. Файлы сюда только добавляются, никогда не
   редактируются и не удаляются без явного указания пользователя.
2. **wiki/** — компилированное знание. Полностью принадлежит LLM: создание,
   обновление, кросс-ссылки, устранение противоречий.
3. **CLAUDE.md** — схема. Эволюционирует вместе с проектом; при изменении
   конвенций сначала обнови этот файл.

`deliverables/` — отдельный контур: это не знания, а продукты работы. Они пишутся
на основе wiki, финальные версии не переписываются задним числом (новая версия =
новый файл).

## Именование

- Исходники: `raw/YYYY-MM-DD_short-title.ext` (дата получения).
- Страницы wiki: `wiki/<category>/short-title.md` — без дат, страница живая.
- Deliverables: `deliverables/<type>/YYYY-MM-DD_<recipient-or-topic>_v1.md` —
  новая версия увеличивает суффикс `_v2`, `_v3`, старую не трогаем.
- Кросс-ссылки внутри wiki — wikilinks: `[[oxtech-platform]]` (совместимо с Obsidian).

## Формат страницы wiki

```markdown
---
type: source | entity | concept | synthesis
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: [raw/2026-07-20_oxtech-multiagent-platform.pdf]
tags: []
---
# Заголовок

Содержимое. Ссылки на связанные страницы: [[other-page]].
Факты — со ссылкой на источник. Противоречия помечать блоком «⚠️ Противоречие».
```

## Операции

### Ingest (обработка нового источника)
1. Файл появляется в `inbox/` (или прикреплён в чат) → переместить в `raw/`
   с правильным именем. Исходник больше не менять.
2. Прочитать источник, обсудить ключевые тезисы с пользователем.
3. Создать конспект в `wiki/sources/`.
4. Обновить/создать затронутые страницы `entities/` и `concepts/`
   (один источник может затронуть 5–15 страниц).
5. Если новое противоречит старому — пометить «⚠️ Противоречие», не затирать молча.
6. Обновить `index.md`, добавить запись в `log.md`.

### Query (ответ на вопрос)
1. Сначала читать `index.md`, затем релевантные страницы — не raw-исходники,
   если wiki достаточно.
2. Отвечать с указанием страниц/источников.
3. Ценные ответы (сравнения, анализы) сохранять в `wiki/synthesis/` — исследования
   должны накапливаться, а не пропадать в истории чата.

### Deliverable (создание рабочего документа)
1. Содержательную основу брать из wiki; недостающие факты — уточнить у пользователя
   или найти (ingest новых источников).
2. Сохранить в соответствующую папку `deliverables/` по конвенции именования.
3. Полезные наработки (структура документа, формулировки позиционирования)
   отражать в wiki или `templates/`.
4. Запись в `log.md`.

### Lint (регулярная проверка здоровья)
По запросу «сделай линт wiki»: искать противоречия между страницами, устаревшие
утверждения, страницы-сироты без входящих ссылок, упомянутые но не созданные
страницы, битые wikilinks, пробелы в данных. Результат — отчёт + правки.

## index.md и log.md

- `index.md` — каталог по категориям: каждая страница со ссылкой и одной строкой
  описания. Обновляется при каждом инжесте и создании страницы.
- `log.md` — append-only. Формат записи:
  `## [YYYY-MM-DD] ingest|query|deliverable|lint | Краткое название`
  (парсится через `grep "^## \[" log.md | tail -5`).

## Git

Репозиторий: `origin = git@github.com:neonix20b/agents-wiki.git`, ветка `main`.

- **Перед ingest (и любой операцией записи)** — сначала синхронизироваться:
  `git pull --rebase origin main`, затем `git status`. Если есть незакоммиченные
  чужие изменения или конфликты — разобраться и сообщить пользователю, не начинать
  операцию поверх грязного состояния.
- **После каждой операции** (ingest / deliverable / lint / изменение схемы) —
  автоматически закоммитить и запушить: `git add -A && git commit && git push`.
  Один логический шаг = один коммит.
- Формат сообщения коммита — как в log.md: `ingest|query|deliverable|lint|schema: краткое описание`
  (например, `ingest: README ox-ai-workers`).
- Push требует SSH-ключей пользователя, поэтому git-команды выполнять локальным
  шеллом на машине пользователя (в Cowork — через Desktop Commander), а не в песочнице.
- Историю не переписывать (`push --force` запрещён); raw/ в git хранится как есть.

## Правила поведения

- Никогда не редактировать файлы в `raw/`.
- Не создавать страницу, не добавив её в `index.md`.
- Не завершать инжест без записи в `log.md`.
- Даты в содержимом — всегда абсолютные (2026-07-20, не «сегодня»).
- Если структура перестаёт подходить — предложить изменение схемы и обновить
  CLAUDE.md, а не нарушать конвенции молча.
