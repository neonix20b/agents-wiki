## Репозиторий и управление

- Определение: «an open protocol enabling communication and interoperability between opaque
  agentic applications».
- Управление: проект «under the Linux Foundation», «contributed by Google».
- Лицензия: Apache License 2.0. Открыт для контрибуций сообщества.
- Текущая версия спецификации: **v1.0.1, выпущена 28 мая 2026**. Всего 11 релизов.
- Полная спецификация: a2a-protocol.org/latest/specification/

## Заявленные цели протокола

- «Break Down Silos: Connect agents across different ecosystems»
- «Enable Complex Collaboration» между специализированными агентами
- «Promote Open Standards» через развитие силами сообщества
- «Preserve Opacity» — агенты сотрудничают, не раскрывая проприетарную логику

## Принципы проектирования

1. **Simplicity** — опора на HTTP, JSON-RPC и SSE вместо проприетарных стандартов.
2. **Enterprise-grade** — расчёт на production: безопасность и требования комплаенса.
3. **Asynchronous-first** — нативная поддержка длительных задач и разрывов соединения.

## Ключевые понятия

- **Agent Card** — «цифровой паспорт» агента по адресу `/.well-known/agent-card.json`
  (RFC 8615). Содержит: идентификацию (имя, описание, провайдер), адрес сервиса,
  возможности (streaming, push notifications), схемы аутентификации (Bearer, OAuth2),
  навыки (описание задач с режимами ввода-вывода и примерами).
- **Task** — задача с жизненным циклом: подача, обработка, завершение; промежуточные
  статусы доступны по ходу выполнения.
- **Message / Artifact** — сообщения и артефакты как отдельные объекты данных, которые
  агенты производят и потребляют.
- Типы данных: текст, файлы, структурированный JSON (мультимодальность).

## Три режима взаимодействия

1. **Синхронный** — запрос-ответ через `sendMessage`.
2. **Потоковый** — Server-Sent Events, обновления в реальном времени через `sendMessageStream`.
3. **Асинхронный** — push-уведомления для сценариев с разорванным соединением.

## Заявленные преимущества

- **Interoperability across frameworks** — агенты на разных технологиях работают вместе
  без индивидуальных интеграций.
- **Security through opacity** — агенты сотрудничают, при этом их «internal logic, memory,
  or proprietary tools» остаются защищены.
- **Enterprise readiness** — аутентификация, авторизация, мониторинг, длительные операции.
- **Multimodality** — типы контента шире простого текста.

## Стратегии обнаружения агентов

1. **Well-known URI** — карточка по `https://{domain}/.well-known/agent-card.json`.
   Автоматическое обнаружение для публичных или доменно-контролируемых агентов;
   чувствительные карточки требуют аутентификации на эндпоинте.
2. **Curated registries** — централизованные каталоги карточек, запрос по критериям
   (навыки, провайдер). Подходит для корпоративных сред и маркетплейсов с требованиями
   к управляемости. **Спецификация пока не стандартизует API реестра.**
3. **Direct configuration** — заданные заранее адреса через конфиги и переменные окружения.
   Проще, но не даёт гибкости при динамике.

## Безопасность обнаружения

- **Authenticated Extended Cards** — расширенные сведения только авторизованным клиентам.
- **Secure Endpoints** — mTLS, ограничения по IP, OAuth 2.0.
- **Registry Selective Disclosure** — реестр отдаёт карточки в зависимости от прав клиента.
- Рекомендация: избегать статических секретов, использовать динамические учётные данные.
- Кэширование: серверам рекомендованы заголовки `Cache-Control` и `ETag`.

## Позиционирование относительно MCP

MCP связывает модели с инструментами и данными; A2A обеспечивает переговоры между агентами.
В документации отмечено: инкапсуляция агента в простой инструмент «fundamentally limiting»
по сравнению с нативной A2A-коммуникацией, которая сохраняет автономию агента и способность
рассуждать в многошаговых взаимодействиях.
